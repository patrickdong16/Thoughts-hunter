# 思想雷达内容质量控制规范 (v3.0)

> 🔴 **最高优先级**：内容质量 · 来源权威 · 分类准确 · 500字符以上
> 
> 本规范为强制执行标准，任何违反行为都是不可接受的。

---

## 核心质量要求（强制执行）

### 1. 字数要求（双标准机制）

**生成标准（内容创建时）：**
```
⛔ 生成内容必须 ≥700 中文字符
⛔ 确保经过传输后仍能满足验证标准
📍 实现位置：backend/services/ai-analyzer.js line 136
```

**验证标准（后端接收时）：**
```
✅ 后端验证阈值：≥300 中文字符
✅ 留有足够缓冲区应对潜在的传输损耗
📍 实现位置：
   - backend/routes/radar.js POST方法 line 179
   - backend/routes/radar.js PUT方法 line 246
   - backend/scripts/add-item.js line 67
```

**为什么有两个标准？**
- 生成时目标高（700字符）→ 确保内容质量和深度
- 验证时宽松（300字符）→ 避免因传输差异导致失败
- 这样即使有30-50%的"损耗"，内容仍能通过验证

### 2. 来源权威性要求

**必须使用的来源类型：**

| 优先级 | 来源类型 | 示例 |
|--------|----------|------|
| ★★★ | 诺贝尔/图灵/普利策获奖者 | Daron Acemoglu, Daniel Kahneman |
| ★★★ | 顶级大学教授 | MIT, Harvard, Stanford, Oxford |
| ★★★ | 顶级智库 | Brookings, CFR, Chatham House |
| ★★ | 长期深耕领域的知识分子 | Tyler Cowen, Ezra Klein, Sam Harris |
| ★★ | 顶级媒体深度栏目 | Foreign Affairs, The Atlantic, FT |
| ★★ | 知名播客深度访谈 | Lex Fridman (≥2hr), Dwarkesh Patel |
| ★ | 行业领袖的深度观点 | 需有实质性论证，非公关 |

**禁止的来源：**
- ❌ 社交媒体碎片化言论
- ❌ 新闻报道（无深度分析）
- ❌ PR/商业软文
- ❌ 未经验证的自媒体

### 3. 日期要求

```
✅ 来源日期必须 ≤ 当前日期（不能是未来）
✅ 可以使用任何过去的优质内容（不限于2025年）
✅ 经典内容（如20世纪的重要著作/演讲）也是有效的
```

### 4. 分类准确性要求（关键！）

**频段定义 - 严格遵守：**

| 频段 | 核心议题 | 应分配的内容类型 |
|------|----------|------------------|
| T1-T3 | **科技** | AI技术本身、开源vs闭源、科技巨头垄断 |
| P1-P3 | **政治** | 全球化、民主制度、气候政策vs增长 |
| H1-H3 | **历史** | 历史周期、文明冲突、帝国兴衰 |
| Φ1-Φ3 | **哲学** | 自由意志、意识本质、道德相对主义 |
| R1-R2 | **宗教** | 宗教角色、科学与信仰 |
| F1-F2 | **金融** | 加密货币、UBI/金融化 |

**常见错误分类（禁止）：**
- ❌ AI风险 → Φ（应该是 **T**，除非讨论意识问题）
- ❌ AI对就业影响 → H（应该是 **T** 或 **F**）
- ❌ 技术停滞 → H（应该是 **T**）
- ❌ AI政策 → P（只有涉及民主/全球化才是P，否则是 **T**）

**正确分类示例：**
- Ilya Sutskever论规模法则 → **T1** (AI技术路线)
- Peter Turchin论历史周期 → **H1** (历史规律)
- John Mearsheimer论大国竞争 → **P1** (国际政治)
- Sam Harris论意识问题 → **Φ2** (意识本质)
- Ray Dalio论债务周期 → **F1** (金融/经济周期)

## 每日配额和分布（强制执行）

```
⛔ 必须严格遵守以下分布，每次执行前必须检查
```

| 领域 | 代码 | 每日数量 | 说明 |
|------|------|----------|------|

| 政治 | P | **恰好1条** | P1/P2/P3任选其一 |
| 历史 | H | **恰好1条** | H1/H2/H3任选其一 |

| 金融 | F | **恰好1条** | F1/F2任选其一 |

### 执行前检查清单

在生成或调整内容之前，必须验证：

```
□ 当日是否有恰好6条内容？
□ T=1, P=1, H=1, Φ=1, F=1, R=1 是否全部满足？
□ 是否有某个领域>1条？（如有，必须删除多余的）
□ 是否有某个领域=0条？（如有，必须添加）
```

### 禁止的情况

| 错误 | 示例 | 后果 |
|------|------|------|
| ❌ 某领域过多 | T=2, P=1, H=1... | 删除多余的T |
| ❌ 某领域缺失 | T=1, P=1, H=0... | 添加H内容 |
| ❌ 总数不足 | 只有5条 | 检查并补齐缺失领域 |
| ❌ 总数过多 | 有7条 | 检查是否有重复领域 |
